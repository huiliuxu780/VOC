# shadcn UI Skill

## 目的
本文件定义仓库中的 shadcn/ui 组件使用策略，用于指导前端开发代理在实现页面、表单、弹层、后台配置界面时，优先采用统一、可复用、可维护的组件体系。

本文件不替代仓库根目录 `AGENTS.md`。  
若与 `AGENTS.md` 冲突，以 `AGENTS.md` 为准。  
若与当前任务中的明确要求冲突，以当前任务要求为准。

---

## 一、适用范围

当任务涉及以下内容时，应优先参考本文件：

- 管理后台页面
- 表单类页面
- 配置面板
- 弹层 / 浮层
- 选项切换
- 输入与编辑控件
- 可复用 UI 组件封装

---

## 二、核心原则

### 1. 优先复用，不重复造轮子
组件实现优先级如下：

1. 仓库内现有 `@/components/ui/*`
2. shadcn/ui 官方组件模式
3. 必要时在现有组件基础上扩展
4. 只有在前三者都不适合时，才自定义新的基础组件

### 2. 基础组件尽量统一
对于同一类交互能力，不要同时存在两套风格、两套 API、两套状态处理方式。

例如：
- 不要一部分页面用 `@/components/ui/textarea`，另一部分页面又手写一版完全不同的 `<textarea>`
- 不要一部分弹层用统一 `Popover`，另一部分又直接手写浮层结构且风格完全不同

### 3. 扩展优先于重写
如果设计需要定制：
- 优先在 shadcn 组件基础上扩展样式或组合能力
- 不要绕开现有组件体系，直接重写一套视觉和行为都不同的基础组件

---

## 三、推荐默认组件映射

下列场景，默认优先采用对应组件：

- 按钮：`@/components/ui/button`
- 单行输入：`@/components/ui/input`
- 多行输入：`@/components/ui/textarea`
- 复选框：`@/components/ui/checkbox`
- 标签页：`@/components/ui/tabs`
- 浮层：`@/components/ui/popover`
- 对话框：`@/components/ui/dialog`
- 抽屉 / 侧滑：`@/components/ui/sheet`
- 下拉菜单：`@/components/ui/dropdown-menu`
- 选择器：`@/components/ui/select`
- 卡片：`@/components/ui/card`
- 表格：`@/components/ui/table`
- 工具提示：`@/components/ui/tooltip`

如果仓库已有本地封装后的等价组件，优先沿用仓库版本。

---

## 四、Textarea 规则

### 1. 默认实现
多行文本输入默认使用：

```tsx
import { Textarea } from "@/components/ui/textarea"
```

### 2. 不应直接手写原生 `<textarea>` 的场景
默认不要直接写带大量样式 class 的原生 `<textarea>`，除非满足以下任一条件：

- 仓库尚未接入 shadcn/ui 组件体系
- 当前场景有明确的第三方依赖要求
- 当前功能需要特殊行为，且在现有 `Textarea` 基础上无法合理扩展

### 3. 定制方式
需要定制时，优先在现有 `Textarea` 组件基础上扩展（className / variant / 封装组合），避免绕过统一组件层。

---

## 五、表单字段组织

表单字段应尽量保持一致结构与语义，优先采用：

- Label
- Description
- Control
- Error state

如果仓库已有字段封装，优先复用。  
如果仓库没有统一封装，优先参考 shadcn/ui 字段组织方式。

---

## 六、状态与可访问性

实现表单控件时，应优先使用语义化状态，不只做视觉模拟：

- 错误状态：优先使用 `aria-invalid`
- 禁用状态：优先使用 `disabled`
- 说明文案：使用描述性字段结构
- 可访问性属性：保持 label、id、描述信息之间的正确关联

---

## 七、一致性要求

- 不要为相同用途同时保留两套冲突的基础组件
- 不要在新页面中随意跳过仓库现有组件体系
- 新增基础组件时，尽量贴近 shadcn/ui API 与目录结构
- 如需大幅定制，优先“基于现有组件扩展”，而不是“重写同类组件”

---

## 八、管理后台页面补充规则

对于后台表单、配置台、控制台页面：

- 输入类组件保持统一高度、圆角、边框和焦点状态
- Textarea、Input、Select、Checkbox、Tabs、Popover 等保持同一体系
- 避免在页面内写大量一次性输入组件样式代码
- 保持暗黑 SaaS 风格一致：弱边框、克制高亮、统一间距、清晰层级

---

## 九、执行建议

在开发任务中，可按以下顺序快速执行：

1. 先检索仓库 `components/ui` 是否已有可复用组件。
2. 若已有，直接复用并在必要时小范围扩展。
3. 若没有，再按 shadcn/ui 模式新增。
4. 提交前检查同类交互是否风格一致、状态语义是否完整。
